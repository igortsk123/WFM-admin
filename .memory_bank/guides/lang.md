# Язык документации

## Основное правило

**Вся документация в проекте WFM должна быть написана на русском языке.**

Это обеспечивает:
- Единообразие всей технической документации
- Упрощение работы для русскоязычной команды разработки
- Снижение барьера входа для новых участников проекта
- Быстрое понимание бизнес-логики и архитектурных решений

## Что должно быть на русском

### Документация
- README файлы (README.md, TESTING.md, BUILD_INSTRUCTIONS.md и т.д.)
- Документация в `.memory_bank/` (domain, backend, web, mobile, guides)
- Файлы CHANGELOG
- Комментарии к архитектурным решениям
- Описания API в swagger/openapi
- Инструкции по развертыванию

### Код
- Комментарии в коде (особенно в сложных местах)
- Docstrings для функций и классов (Python, Kotlin, Swift)
- JSDoc/TSDoc комментарии (TypeScript/JavaScript)
- Описания TODO и FIXME

### Git
- Описания коммитов могут быть на русском (сам commit message - на английском)
- Описания Pull Request
- Комментарии в issues

## Исключения (английский допустим)

### Код
- Названия переменных, функций, классов, методов
- Названия файлов и директорий
- URL и пути к файлам
- Команды терминала

### Технические термины
- Устоявшиеся термины без прямого перевода
  - Memory Bank
  - Pull Request
  - State Machine
  - MVVM, DI, API и другие аббревиатуры

### Конфигурация
- Файлы конфигурации (docker-compose.yml, package.json и т.д.)
- Environment variables
- Ключи в JSON/YAML

### Комментарии в коде

✅ **Правильно (Python):**
```python
def validate_task_transition(current_state: TaskState, new_state: TaskState) -> bool:
    """
    Проверяет допустимость перехода между состояниями задачи.

    Правила переходов определены в `.memory_bank/domain/task_states.md`
    """
    allowed_transitions = {
        TaskState.NEW: [TaskState.IN_PROGRESS],
        TaskState.IN_PROGRESS: [TaskState.PAUSED, TaskState.COMPLETED],
        TaskState.PAUSED: [TaskState.IN_PROGRESS],
        TaskState.COMPLETED: []
    }
    return new_state in allowed_transitions.get(current_state, [])
```

✅ **Правильно (Swift):**
```swift
/// Проверяет возможность перехода к новому состоянию
/// - Parameter newState: Целевое состояние для перехода
/// - Returns: true если переход разрешён
func canTransition(to newState: TaskState) -> Bool {
    return allowedTransitions.contains(newState)
}
```

## Миграция существующей документации

Если в проекте есть документация на английском языке:

1. **Приоритет миграции:**
   - Сначала: README, Memory Bank документы
   - Потом: Комментарии в критичном коде
   - В последнюю очередь: Остальные комментарии

2. **Постепенный переход:**
   - Новая документация пишется только на русском
   - Старая документация обновляется по мере необходимости
   - При рефакторинге кода - обновляем комментарии

3. **Смешанный код:**
   - Если часть кода на английском, новые комментарии все равно на русском
   - Постепенно приводим к единообразию

## Инструменты для проверки

### Автоматизация (будущее)
- Pre-commit hook для проверки языка в README файлах
- Linter для проверки docstring на русском языке
- CI проверка документации в `.memory_bank/`

### Ручная проверка
- Code review обращает внимание на язык документации
- Pull Request не мержится с английской документацией в новых файлах

## Почему русский язык

1. **Команда разработки** - основная команда русскоязычная
2. **Бизнес-контекст** - проект для российского рынка
3. **Доменная модель** - бизнес-логика понятнее на родном языке
4. **Скорость разработки** - не нужно переводить в уме при чтении
5. **Онбординг** - новые разработчики быстрее входят в проект

## Контакт

При возникновении вопросов о языке документации - обращайтесь к тимлиду или используйте обсуждения в Pull Request.
