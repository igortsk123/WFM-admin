# Руководство по Firebase Analytics для аналитиков

Этот документ поможет дизайнеру или аналитику самостоятельно находить нужные данные и строить воронки в Firebase Console, не обращаясь к разработчикам.

Все события, упомянутые здесь, описаны в `mobile_events.md`.

---

## Доступ

1. Открыть [Firebase Console](https://console.firebase.google.com/)
2. Выбрать проект WFM
3. В левом меню перейти в **Analytics**

---

## Где смотреть события

**Analytics → Events** — список всех зарегистрированных событий, их количество и динамика за период.

Нажав на событие, можно увидеть распределение по параметрам. Например, для `task_detail_viewed` — сколько раз открывали задачи в каждом `task_state`.

**Analytics → DebugView** — режим реального времени для проверки новых событий во время разработки. Не использовать для анализа продакшн-данных.

---

## Сегментация по роли

Все пользователи разделены на два сегмента через user property `user_role`:
- `worker` — сотрудники магазина
- `manager` — управляющие

Чтобы применить сегмент: в любом отчёте нажать **Add filter → User property → user_role**.

---

## Идентификация конкретного пользователя

Firebase хранит `userId` = внутренний `user_id` из БД проекта.

**Как найти пользователя:**
1. В Firebase: **Analytics → User Explorer** → ввести `userId` в поиск
2. В БД проекта: запрос к Users API `GET /users/{user_id}` или в таблицу `users` по полю `id`

Это позволяет, например, проверить путь конкретного пользователя после обращения в поддержку.

## Воронки

Воронки строятся в **Analytics → Funnels** (или через **Explore → Funnel exploration**).

### Воронка авторизации

```
phone_input_viewed
→ phone_submitted
→ code_input_viewed
→ code_submitted
→ login_completed
```

Показывает, на каком шаге пользователи отваливаются при входе.

### Воронка выполнения задачи (работник)

```
task_detail_viewed
→ task_start_tapped
→ task_complete_sheet_opened
→ task_complete_submitted
```

Показывает конверсию от открытия задачи до её завершения.

### Воронка проверки задачи (управляющий)

```
task_review_sheet_opened
→ task_approved_tapped | task_rejected_tapped
```

Показывает соотношение принятых и отклонённых задач.

---

## Полезные отчёты

**Retention** (`Analytics → Retention`) — как часто пользователи возвращаются в приложение после первого входа.

**User Explorer** (`Analytics → User Explorer`) — путь конкретного пользователя: список всех событий в хронологическом порядке.

**Events over time** — выбрать событие в `Analytics → Events`, нажать на него и изучить динамику по дням/неделям.

---

## Ограничения Firebase

- Данные появляются с задержкой ~24 часа (кроме DebugView)
- Бесплатный тариф (Spark) имеет ограничения на количество аудиторий и Funnel шагов
- Исторические данные доступны за последние 14 месяцев по умолчанию

## Мониторинг производительности сети

Событие `api_request_completed` трекается автоматически для каждого HTTP-запроса и предназначено для технического мониторинга — не для поведенческих воронок.

**Полезные запросы в Firebase:**
- **Среднее время ответа по эндпоинту:** `path` + средний `duration_ms`
- **Доля ошибок по магазину:** `store_id` + `is_error = true`
- **Деградация после деплоя:** `duration_ms` по времени

**Важно:** `is_error = false` для HTTP 4xx (401, 409 и т.д.) — это прикладные ошибки, не инфраструктурные. `is_error = true` только для 5xx, таймаутов и сетевых ошибок.

## Что делать, если нужного события нет

Если в Firebase нет нужного события — скорее всего, оно ещё не реализовано. Нужно:
1. Проверить `mobile_events.md` — есть ли событие в реестре
2. Если есть — обратиться к разработчику, возможно событие ещё не реализовано
3. Если нет — описать потребность, разработчик добавит событие в реестр и реализует
