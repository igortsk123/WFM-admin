# Правила аналитики для разработчиков

## Обязательные события

Каждый значимый экран обязан отправлять события жизненного цикла и событие появления. Значимый экран — любой экран с данными или действием пользователя (не промежуточный контейнер навигации).

**Обязательно для каждого экрана:**
- `*_viewed` — экран отображён (аналог `onAppear` / `LaunchedEffect`)

**Обязательно для важных действий:**
- Нажатие на основную CTA-кнопку (начать задачу, завершить, подтвердить)
- Отправка формы (авторизация, создание задачи)
- Применение фильтра

**Не нужно** отслеживать каждый тап — только действия с бизнес-смыслом.

---

## Запрещённые данные (приватность)

❌ **Никогда не передавать в аналитику:**

- ФИО, дата рождения, паспортные данные, СНИЛС, ИНН
- Номер телефона в исходном виде
- Адрес и точная геолокация
- JWT-токены, session ID, внутренние ключи
- Содержимое отчётов работников (`report_text`, тексты комментариев)
- URL-адреса фотографий (`report_image_url`)

✅ **Разрешено:**
- Внутренний `user_id` (целое число) — для идентификации пользователя
- Роль пользователя: `"worker"` / `"manager"`
- UUID задачи — обезличен, не несёт персональных данных
- Булевы флаги: `has_photo`, `has_comment`, `requires_photo`
- Числовые метрики: `tasks_count`, `planned_minutes`

## Идентификация пользователя

После успешного входа устанавливается `userId` и роль через `analyticsService.setUser(userId:role:)`. Этот вызов одновременно устанавливает идентификатор в Firebase и сохраняет его в `SemetricsAnalyticsService` для передачи в каждое событие.

```
userId    = user_id из Users API (целое число, например 42)
user_role = "worker" | "manager"
```

**Firebase:** userId + user property `user_role`
**Semetrics:** userId передаётся в поле `user_id` каждого события

При выходе из приложения `userId` сбрасывается в обоих провайдерах через `analyticsService.resetUser()`.

## Именование событий

- Формат: `snake_case`
- Шаблон: `{объект}_{действие}` — например `task_start_tapped`, `tasks_list_viewed`
- Для экранов: суффикс `_viewed` — например `home_viewed`, `settings_viewed`
- Для нажатий: суффикс `_tapped` — например `logout_tapped`
- Для системных событий: суффикс `_completed` — например `login_completed`

## Процесс добавления нового события

1. **Сначала обнови `mobile_events.md`** — добавь событие в реестр
2. Добавь константу в `AnalyticsEvent` (enum/sealed class)
3. Добавь маппинг в **каждый** аналитический сервис:
   - iOS: `FirebaseAnalyticsService` + `SemetricsAnalyticsService`
   - Android GMS: `FirebaseAnalyticsService` (src/gms) + `SemetricsAnalyticsService`
   - Android HMS: `AppMetricaAnalyticsService` (src/hms) + `SemetricsAnalyticsService`
4. Вызови `analyticsService.track(event)` в нужном месте
5. Проверь в Firebase DebugView / AppMetrica, что событие приходит с правильными параметрами

> ❗ Нельзя добавлять событие в код, не добавив его в реестр `mobile_events.md`. Реестр — источник правды.
> ❗ При добавлении нового события маппинг нужно добавить во ВСЕ три сервиса (Firebase GMS, AppMetrica HMS, Semetrics).

---

## Параметры событий

- Передавай только параметры, которые реально нужны для анализа
- Не добавляй параметры «на будущее» — лучше добавить позже, когда возникнет потребность
- Firebase ограничивает: 25 параметров на событие, имена до 40 символов, строки до 100 символов

---

## Антипаттерны

| Нельзя | Почему |
|---|---|
| Передавать телефон или ФИО | Нарушение приватности |
| Добавлять событие без реестра | Теряется документация |
| Трекать каждый скролл или тап | Засоряет аналитику |
| Дублировать одно событие из разных мест кода | Искажает метрики |
| Использовать SDK Firebase или Semetrics напрямую, минуя `AnalyticsService` | Ломает fan-out и возможность смены провайдера |
