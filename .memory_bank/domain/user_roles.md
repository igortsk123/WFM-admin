# Domain Model: User Roles and Permissions

## Зачем нужна система ролей

Система ролей в WFM решает ключевую задачу — **разделение ответственности** между управляющими магазинами и работниками.

**Проблема без ролей:**
- Все пользователи видят все задачи → путаница и риск выполнения чужих задач
- Нет разделения прав на создание задач и их выполнение → хаос в управлении
- Невозможно отследить, кто выполнил задачу и кто её назначил → отсутствие ответственности

**Что даёт система ролей:**
- **Чёткое разделение прав:** управляющий управляет, работник выполняет
- **Фокус на своих задачах:** работник видит только то, что ему назначено
- **Гибкость через привилегии:** один работник может совмещать кассу и склад, другой — только выкладку
- **Контроль качества:** управляющий проверяет отчёты и подтверждает выполнение задач

---

## Обзор

Система ролей WFM построена на упрощённой модели с **двумя ролями** и системой **привилегий** для работников.

**Ключевые принципы:**
- Роль определяется **через должность (Position)**, а не напрямую у пользователя
- У каждой должности есть `role_id` (по умолчанию worker), роль определяется через цепочку: `User → Assignment → Position → Role`
- Роли **взаимоисключающие** — нельзя быть одновременно управляющим и работником
- Внутри роли WORKER есть **привилегии** (permissions), которые назначает управляющий
- Пользователь без назначения (assignment) не имеет роли
- Магазин пользователя определяется через `assignments.store` (вложенный объект)

---

## Роли

### MANAGER (Управляющий)

**Основная функция:** Управление магазином, персоналом и задачами

**Как стать управляющим:** Администратор вручную устанавливает `role_id = 2` (manager) у должности в таблице `positions`

**Возможности:**
- Создание задач (плановых и дополнительных)
- Назначение задач работникам
- Управление привилегиями работников
- Просмотр всех задач магазина
- Проверка отчётов по задачам
- Подтверждение или отклонение выполнения задач
- Просмотр аналитики магазина

**Видимость:**
- Видит **ВСЕ задачи своего магазина** (всех работников)
- Видит все отчёты по задачам
- Видит список работников и их привилегии

**Права:**
- Создание, редактирование, удаление задач
- Назначение исполнителей
- Назначение и удаление привилегий работникам
- Требование отчёта по любой задаче
- Подтверждение/отклонение отчётов
- Просмотр аналитики план/факт

---

### WORKER (Работник)

**Основная функция:** Выполнение задач в магазине

**Как стать работником:** По умолчанию — все должности имеют `role_id = 1` (worker)

**Возможности:**
- Просмотр назначенных задач
- Выполнение задач (start/pause/complete)
- Предоставление отчётов по задачам
- Просмотр инструкций

**Видимость:**
- Видит **только свои задачи** (assignee_id = user_id)
- Не видит задачи других работников
- Видит свою историю выполнения

**Права:**
- Выполнение задач (переходы между состояниями)
- Предоставление отчётов (текст, фото, метаданные)
- Просмотр инструкций по задачам

**Привилегии:**
Работник имеет набор привилегий (permissions), которые назначает управляющий:
- **CASHIER** — работа на кассе
- **SALES_FLOOR** — работа в торговом зале (выкладка, переоценка, уборка)
- **SELF_CHECKOUT** — работа на кассе самообслуживания
- **WAREHOUSE** — работа на складе (приёмка, инвентаризация)

Привилегии могут расширяться в будущем.

**Что определяют привилегии:**
- Какие типы задач работник может выполнять
- Доступ к определённым зонам магазина
- Видимость инструкций по определённым операциям

---

### NO_ROLE (Без роли / без назначения)

**Состояние:** Пользователь зарегистрирован, но нет назначения (assignment) с должностью

**Видимость:**
- Не видит задачи
- Видит экран "Ожидание назначения роли"

**Права:**
- Просмотр своего профиля
- Обновление контактных данных

**Переход:**
- После синхронизации с LAMA создаётся assignment с должностью → пользователь получает роль через position.role_id

## Модели данных

### User

```
id: INTEGER (первичный ключ, auto-increment, внутренний ID)
sso_id: UUID (уникальный, NOT NULL — UUID из SSO/JWT, поле 'u' в токене)
external_id: INTEGER | null (employee_id из LAMA)
type_id: INTEGER | null (FK на EmployeeType)
updated_at: datetime
updated_by: INTEGER | null (внутренний int ID пользователя, внесшего изменения)
```

> **Примечание:** Роль и магазин определяются через assignments. `role_id` и `store_id` убраны из таблицы users.

---

### Position (Должность)

Справочник должностей. Каждая должность имеет привязку к роли.

```
id: INTEGER (автоинкремент)
code: string (уникальный код должности)
name: string (название должности)
description: string | null
role_id: INTEGER (FK на Role, NOT NULL, default=1 — worker)
```

**Важно:** По умолчанию все должности имеют `role_id = 1` (worker). Для должностей с ролью управляющего администратор вручную устанавливает `role_id = 2` (manager).

---

### Role (Роль)

Справочник ролей:
- `id=1, code="worker"` — работник
- `id=2, code="manager"` — управляющий

---

### WorkerPermission

```
id: UUID
user_id: INTEGER (внешний ключ на users.id, CASCADE DELETE)
permission: PermissionType (CASHIER | SALES_FLOOR | SELF_CHECKOUT | WAREHOUSE | ...)
granted_at: datetime
granted_by: INTEGER (внутренний int ID управляющего, назначившего привилегию)
revoked_at: datetime | null (если отозвана)
```

**Ограничения:**
- Привилегии могут быть назначены только пользователям с ролью WORKER
- Управляющий может назначать привилегии только работникам своего магазина
- Привилегия может быть отозвана (soft delete через `revoked_at`)

---

### Assignment (Назначение сотрудника)

Модель для хранения назначений сотрудника — связей между работником, должностью, разрядом и магазином. Данные синхронизируются из LAMA. У одного пользователя может быть несколько назначений (работа в разных магазинах).

```
id: INTEGER (автоинкремент)
user_id: INTEGER (внешний ключ на users.id, CASCADE DELETE)
external_id: INTEGER | null (employee_in_shop_id из LAMA, unique)
company_name: string | null (название компании)
position_id: INTEGER | null (FK на Position)
rank_id: INTEGER | null (FK на Rank)
store_id: INTEGER | null (FK на Store, только в БД — в API-ответе не возвращается)
store: Store | null (объект магазина, вложен в API-ответ)
date_start: date | null (дата начала назначения)
date_end: date | null (дата окончания назначения)
```

**Связь с LAMA:**
- `external_id` соответствует `employee_in_shop_id` из ответа LAMA `/employee/`
- Используется для запроса смен (`GET /shift/?employee_in_shop_id={external_id}`)
- Должность (`position_id`) и разряд (`rank_id`) берутся из данных LAMA

**Определение роли и магазина:**
- Роль пользователя: `Assignment → Position → Role`
- Магазин пользователя: `Assignment.store` (вложенный объект `Store` с полями `id`, `name`, `address`)

**Логика выбора текущего assignment (для пользователей с несколькими назначениями):**
- При загрузке пользователя проверяется сохраненный `selectedAssignmentId` в локальных преференсах
- Если сохраненный ID найден в списке — используется он
- Если не найден или не сохранен — берется первый assignment и его ID сохраняется
- Текущий выбранный assignment хранится в `UserManager.currentAssignment`
- Во всех UI компонентах используется `currentAssignment`, а не `assignments.first()`

> **Примечание:** Ранее `role_id` и `store_id` хранились непосредственно в модели User. Теперь роль определяется через должность (Position.role_id), а магазин — через назначение (Assignment.store).

---

### Rank (Разряд)

Справочник разрядов сотрудников. Создаётся автоматически при синхронизации с LAMA.

```
id: INTEGER (автоинкремент)
code: string (уникальный код разряда)
name: string (название разряда)
```

## Бизнес-правила

### Правило 1: Роль определяется через должность
- Роль пользователя определяется через цепочку: `User → Assignment → Position → Role`
- По умолчанию все должности имеют роль worker (`role_id = 1`)
- Для должностей управляющих администратор вручную устанавливает `role_id = 2`

### Правило 2: Управляющий не имеет привилегий
- Если роль через должность = MANAGER, у пользователя не может быть записей в таблице `WorkerPermission`
- Управляющий имеет полный доступ ко всем функциям без привилегий

### Правило 3: Привилегии только для работников
- Привилегии могут быть назначены только пользователям с ролью WORKER (определяемой через должность)
- Попытка назначить привилегию управляющему возвращает ошибку

### Правило 4: Управляющий управляет привилегиями
- Только управляющий может назначать и отзывать привилегии
- Управляющий может назначать привилегии только работникам своего магазина

### Правило 5: Привилегии не блокируют задачи
- Работник может получать задачи **любого типа** независимо от привилегий
- Привилегии — это информация для управляющего о компетенциях работника
- Управляющий сам решает, назначать ли задачу работнику без соответствующей привилегии

### Правило 6: Без назначения — нет доступа к задачам
- Пользователь без assignment не имеет роли и не видит задачи
- После синхронизации с LAMA создаётся assignment → появляется роль и доступ

### Правило 7: Магазин через assignment
- Магазин пользователя определяется через `assignments.store` (вложенный объект)
- У пользователя может быть несколько назначений в разных магазинах

## Итог

**Ключевые моменты:**
- 2 роли: MANAGER и WORKER, определяются через `Position.role_id` (по умолчанию worker)
- Цепочка определения роли: `User → Assignment → Position → Role`
- Магазин пользователя: `Assignment.store` — вложенный объект Store (store_id хранится в БД, но в API-ответе не возвращается)
- Привилегии назначаются только работникам (CASHIER, SALES_FLOOR, SELF_CHECKOUT, WAREHOUSE)
- Привилегии не блокируют назначение задач — это индикаторы компетенций для управляющего
- Управляющий видит все задачи магазина, работник — только свои
- Пользователь без назначения не имеет доступа к задачам
- Назначения (Assignment) хранят должность, разряд и привязку к магазину (синхронизируются из LAMA)
- `User.external_id` — INTEGER, employee_id из LAMA
