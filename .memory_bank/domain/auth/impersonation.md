# Impersonation — «Войти как»

Функция для ограниченного круга разработчиков: возможность просматривать приложение от имени любого пользователя, не зная его пароля.

**Связанные документы:**
- `.memory_bank/domain/auth/auth_flow.md` — основной flow авторизации
- `.memory_bank/backend/apis/api_users.md` — internal endpoint `GET /internal/id-by-phone`
- `.memory_bank/mobile/feature_settings/settings_screen.md` — UI кнопки «Войти как»
- `.memory_bank/mobile/architecture/networking.md` — хидер `X-Auth-By`

---

## Концепция

Разработчик с флагом `dev` в JWT может войти под другим пользователем, введя его номер телефона. После этого все запросы к API выполняются от имени этого пользователя — backend возвращает те же данные, что видит настоящий пользователь.

**Назначение:** отладка пользовательского контекста без доступа к паролям и без изменения данных.

---

## Механизм

### JWT флаг доступа

JWT от Beyond Violet содержит поле `flags` — словарь булевых значений:

```json
{ "flags": { "dev": true } }
```

Только пользователи с `flags.dev = true` имеют доступ к функции. Это поле устанавливается на стороне SSO и не может быть изменено клиентом.

### Хидер X-Auth-By

После ввода номера телефона приложение добавляет хидер к каждому HTTP запросу:

```
X-Auth-By: +79001234567
```

### Проверка на сервере

1. Backend расшифровывает JWT текущего пользователя
2. Проверяет наличие `flags.dev = true` — без этого хидер игнорируется
3. Если флаг есть — ищет пользователя по телефону в БД
4. Возвращает `user_id` найденного пользователя вместо user_id владельца токена

**Безопасность:** хидер без `flags.dev` полностью игнорируется. Злоумышленник не может использовать `X-Auth-By` без специального JWT.

---

## Правила

- Функция доступна **только** пользователям с `flags.dev = true` в JWT
- `flags.dev` устанавливается в Beyond Violet SSO — WFM не управляет этим флагом
- Если пользователь с указанным телефоном не найден — запрос выполняется как обычно (от имени владельца токена)
- Номер телефона хранится **только локально** на устройстве разработчика (UserDefaults / DataStore)
- Функция не оставляет следов в данных целевого пользователя — только чтение

---

## Отображение в UI

Кнопка «Войти как» показывается в экране профиля (Settings) **только** если `flags.dev = true` в текущем JWT.

Декодирование JWT на клиенте — только Base64 payload, без верификации подписи (подпись проверяет сервер).

При нажатии кнопки: диалог с полем ввода номера телефона. Кнопка «Очистить» — сбрасывает impersonation и возвращает к собственному аккаунту.

Текущий режим impersonation отображается в профиле (например, плашка с номером телефона).

## Платформы

| Платформо | Хранение номера | JWT декодирование |
|-----------|----------------|-------------------|
| iOS | `UserDefaults` ключ `impersonation_phone` | Base64 decode payload, ручной парсинг |
| Android | `DataStore Preferences` ключ `impersonation_phone` | Base64 decode payload, ручной парсинг |

### iOS: добавление хидера

В `APIClient.swift` в методе формирования запроса, после добавления общих хидеров:

```swift
// Только если номер сохранён
if let phone = ImpersonationStorage.shared.phone, !phone.isEmpty {
    request.setValue(phone, forHTTPHeaderField: "X-Auth-By")
}
```

### Android: добавление хидера

В `ApiClient.kt` в методе `addCommonHeaders()`:

```kotlin
ImpersonationStorage.getPhone(context)?.takeIf { it.isNotBlank() }?.let {
    headers.append("X-Auth-By", it)
}
```
