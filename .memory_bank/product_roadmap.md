# WFM — Продуктовый роадмап и архитектурные импликации

> Источник: Product Owner, март 2026. Цель документа в Memory Bank —
> чтобы AI-агент понимал КУДА идёт система и принимал архитектурные решения,
> которые не блокируют следующие этапы.

---

## Большая идея

Заменить Telegram-бот на **операционную систему магазина** — замкнутый цикл управления:

```
ДАННЫЕ МАГАЗИНА → AI-анализ → СТРАТЕГИЧЕСКИЕ ПРИОРИТЕТЫ
→ ГЕНЕРАЦИЯ ЗАДАЧ → ВЫПОЛНЕНИЕ → МОТИВАЦИЯ → СОЦИАЛЬНАЯ ДИНАМИКА → НОВЫЕ ДАННЫЕ
```

Это движение через 6 слоёв продукта:

| Слой | Суть |
|------|------|
| **Стратегия** | Топ-менеджмент задаёт приоритеты через «паутинку» (10 очков по осям) |
| **AI и аналитика** | Превращает данные в задачи и подсказки сотрудникам |
| **Система задач (ядро)** | Постановка, контроль, подтверждение — то, что строится сейчас |
| **Мотивация и экономика** | Бонусные задачи, коэффициенты, начисления |
| **Социальная динамика** | Рейтинги, стрики, командные челленджи |
| **Платформа знаний** | Tacit knowledge, микро-обучение, наставники |

---

## Роадмап: 7 микропроектов (M0–M6)

Каждый микропроект — 1–2 месяца, создаёт ценность сам по себе, открывает следующий.

```
Apr 2026        May             Jun–Jul         Aug–Sep         Oct–Dec
────────────────────────────────────────────────────────────────────────
FOUNDATION      M0 Mobile MVP
                (сейчас)
VALUE                           M1 AI Coach     M2 Smart Verif.
ALIGNMENT                       M3 Strategy Radar (паутинка)
ENGAGEMENT                                      M4 Social Dynamics
INCENTIVES                                                      M5 Bonus Tasks
                                                                M6 KPI Payouts
```

**Гейты (контрольные точки):**
- **Gate A** (после M0): приложение активно используют менеджеры + telemetry >95% → открывает AI/KPI/social
- **Gate B** (после M1/M2): снизился return rate + нагрузка на проверку → масштабируем rollout
- **Gate C** (после M3/M4): стратегические оси приняты + вовлечённость растёт → бюджет на бонусы
- **Gate D** (после M5): экономика очков меняет поведение → интеграция с зарплатой

---

## M0 — Mobile MVP + Event Telemetry (СЕЙЧАС)

**Цель:** Заменить Telegram. Собирать чистые поведенческие события.

**Что строим сейчас — критично для всего будущего:**
- Task inbox для менеджера (swipe accept/decline, быстрые фильтры)
- Push-уведомления (новые задачи, просрочки, нужна проверка)
- Карточка задачи: чек-листы, фото, история статусов
- Быстрые режимы верификации (full / sampling / photo-required per task type)
- **Event log** — кто, когда, что сделал с задачей

**Метрики успеха:** Manager DAU, % задач обработано с мобильного, event completeness >95%.

---

## M1 — AI Task Coach (Value layer)

**Цель:** AI-подсказки к каждой задаче → меньше возвратов, быстрее онбординг.

**Зависит от:** M0 event telemetry + taxonomy work_types.

**UX:** Блок "AI helper" на карточке задачи: «Зачем важно», «3 шага», «Частые ошибки».
Кнопка «Полезно / не полезно» для обучения.

**Метрики:** Return rate ↓, time-to-complete у новичков ↓, AI-helpful rate >60%.

---

## M2 — Smart Verification (Risk-based Sampling)

**Цель:** Разгрузить менеджера — верифицировать только то, что важно.

**Зависит от:** M0 events; опционально M1.

**Логика:** Risk score = f(тип задачи, история исполнителя, аномалии магазина).
Очередь проверки: High/Med/Low. Фото/гео только для High.

**Метрики:** Время менеджера на проверку ↓, defect rate стабильный или ↓.

---

## M3 — Strategy Radar (10-point Spider → Task Quotas)

**Цель:** Топ-менеджмент задаёт фокус → система конвертирует в квоты задач.

**Зависит от:** M0 taxonomy (work_types, zones).

**UX:** «Паутинка» — распредели 10 очков по осям (чистота, выкладка, скорость...).
Система переводит в квоты задач на неделю по магазинам.

**Метрики:** Доля задач по топ-2–3 осям ↑, «шумовые» задачи ↓.

---

## M4 — Social Dynamics (Ratings + Challenges)

**Цель:** Немонетарная мотивация: прогресс, рейтинги, командные челленджи.

**Важно:** Этичная механика — без «кнута». Упор на команду, не на индивида.

**Метрики:** Своевременность выполнения задач ↑, DAU сотрудников ↑.

---

## M5 — Bonus Task Constructor (Points Economy)

**Цель:** Бонусные задачи с коэффициентами оплаты, лимиты нагрузки, очки без денег.

**Аналог:** Uber surge pricing для труда.

**Метрики:** Uptake rate бонусных задач, покрытие «дыр» в расписании.

---

## M6 — KPI Payout Pipeline

**Цель:** Конвертация очков → выплаты как переменная часть зарплаты.

**Зависит от:** M5 points ledger + финансово-юридическое согласование.

## Архитектурные импликации для разработки

> Это — ключевая секция. Объясняет, почему нельзя срезать углы в текущем M0.

### `task_events` — фундамент всего

Таблица `task_events` — это **операционная телеметрия** всей системы.
Без полного event log невозможны: KPI (M5/M6), AI coaching (M1), risk scoring (M2), leaderboards (M4).

**Правило: при любом изменении состояния задачи — ВСЕГДА писать в task_events.**
Пропущенное событие невосстановимо и блокирует все аналитические фичи.

Минимально необходимые события сейчас:
`task_view`, `start`, `pause`, `resume`, `complete`, `approve` (`verify`), `reject` (`return`).
В будущем добавятся: `decline_reason`, `return_reason`, `ai_feedback`.

### `work_types` + `zones` + `categories` — оси стратегии (M3)

Taxonomy задач — это будущие «оси паутинки» в M3 Strategy Radar.
Нельзя допускать дубли, опечатки, несогласованные названия.
При добавлении новых типов работ — проверять, что они входят в существующую таксономию.

### `acceptance_policy` + `requires_photo` — семена M2

Уже реализованные поля `acceptance_policy` (AUTO/MANUAL) и `requires_photo` —
это v0 risk-based verification. M2 расширит их до полноценного risk score.
Не удалять, не игнорировать при разработке новых флоу.

### Модель данных должна поддерживать будущее

| Будущая фича | Что потребуется в БД |
|---|---|
| M1 AI Coach | `ai_hint` (per task_type, версионирован), feedback events |
| M2 Risk Scoring | `risk_score` на задаче, `performer_return_rate` на пользователе |
| M3 Strategy | `strategy_axes` table, `task_quota` per store/week |
| M4 Leaderboards | `user_score`, `user_streak`, materialized view по магазинам |
| M5 Bonus | `bonus_coefficient`, `points_earned` на задаче, `points_ledger` |
| M6 Payouts | `payout_register`, `payout_period`, reconciliation |

**Принцип:** Не проектировать эти таблицы заранее, но не принимать решений,
которые делают их добавление болезненным (например, не хардкодить логику в код вместо БД).

### Партнёры (non-LAMA клиенты)

Малый бизнес (кафе, небольшой ритейл) использует ту же систему задач без LAMA-интеграции.
Партнёры — это пространство имён: `partners` → `stores` → `assignments` → users.
Все фичи роадмапа должны работать для всех партнёров, а не только для LAMA-клиентов.

---

## Топ-5 гипотез (80% эффекта)

По оценке команды, максимальный impact дадут:

1. **AI-подсказки** (M1) — снижение возвратов, скорость онбординга
2. **Умный контроль задач** (M2) — разгрузка менеджера
3. **Паутинка приоритетов** (M3) — связь операций со стратегией
4. **Рейтинги** (M4) — изменение поведения сотрудников
5. **Бонусные задачи** (M5) — закрытие кадровых дыр в пиковую нагрузку
