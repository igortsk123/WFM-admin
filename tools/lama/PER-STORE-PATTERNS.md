## Findings

1. **zone_affinity** доминирует (44.6%) — это primary signal: директор предпочитает сотрудника с наибольшим past-experience по этой metric.
2. **work_type_affinity** — secondary (40.5%). Часто confounded с первым (zone и work_type сильно коррелируют — кто часто выкладывает Бакалею знает Выкладку).
3. **rank/seniority** — 11.7% — слабый сигнал отдельно.
4. **Administrators** часто берут на себя задачи (31.7%) — возможно нужен бонус роли в scoring.
5. **balance** (12.0%) — значимый отрицательный signal: директор НЕ балансирует загрузку первым приоритетом — концентрирует работу.

## Предложение для iter#2 (scoring weights)

Пропорционально aggregate match rates, нормировано к 100%:

```
  zone_affinity            :  41.0%
  work_type_affinity       :  37.2%
  balance_load             :  11.0%
  rank_seniority           :  10.8%
```

Эти weights — отправная точка. На практике иter#2 сначала фильтрует по constraint'ам (зона/work_type whitelist), потом применяет weighted score:

```
score(emp, task) = 
    0.41 * zone_affinity_norm(emp, task.zone)
  + 0.37 * work_type_affinity_norm(emp, task.wtype)
  + 0.11 * (1 - load_norm(emp))
  + 0.11 * rank_norm(emp)
```

`*_norm` — minmax-normalization среди eligible candidates для этого task. `load_norm` инвертирован (low load = high score), но weight `0.11` минимален — балансировка не доминирует, как реальность показывает.
