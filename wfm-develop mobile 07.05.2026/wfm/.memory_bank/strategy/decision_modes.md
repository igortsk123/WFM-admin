# Режимы работы над задачами (FAST / STANDARD / FULL)

Документ формализует уже неявную норму проекта: **не любая задача требует одинаковой глубины анализа**. Цель — не запускать тяжёлый процесс на мелкой правке и не делать без него миграции.

**Связанные документы:**
- `.memory_bank/strategy/ai_oversight_policy.md` — обязательный чеклист для AI-фич (часть FULL-режима)
- `.memory_bank/strategy/decision_record.md` — формат ADR (обязателен в FULL, опционален в STANDARD)
- `.memory_bank/strategy/theoretical_foundation.md` — почему «не срезать» данные (FULL-режим часто про защиту лоуэр-слоёв)

---

## Принцип

Режим выбирается **до** начала работы и определяет:
- нужно ли составлять `Plan` (через Plan-tool Claude Code),
- нужно ли вести Decision Record,
- нужно ли проходить чеклист `ai_oversight_policy.md`.

Сомнения → выбираем более тяжёлый режим. Дешевле переплатить процессом, чем починить срезанное поле в БД.

---

## FAST

**Когда:** локальная, обратимая правка без новой логики.

**Признаки:**
- правка текста, цвета, отступа, иконки;
- мелкий баг-фикс без изменения контракта;
- рефакторинг без поведенческих изменений;
- никакой новой колонки/эндпоинта/события.

**Что НЕ делаем:** не запускаем `Plan`, не пишем Decision Record, не зовём Skeptic-перспективу.
**Что делаем:** правим, проверяем, коммитим.

---

## STANDARD

**Когда:** новая фича или изменение поведения в пределах существующих контрактов.

**Признаки:**
- новый экран/эндпоинт по существующей модели данных;
- новое поле в DTO без миграции БД (или с тривиальной миграцией);
- изменение UI-логики, но без новой роли/привилегии;
- добавление аналитического события (см. `.memory_bank/analytics/rules.md`).

**Что делаем:**
- `Plan` — да, если задача затрагивает > 1 файла или > 1 платформу;
- Decision Record — нет, если решение очевидное; да, если был выбор между альтернативами;
- проверка `documentation_style.md` при правке Memory Bank.

---

## FULL

**Когда:** изменение, которое влияет на структуру системы, межсервисный контракт или AI-границу.

**Признаки (любой пункт):**
- миграция БД с новой колонкой/таблицей;
- изменение межсервисного контракта (`/internal/*`, формат событий, WS-протокол);
- новая роль или привилегия;
- любое появление AI/ML в продуктовом флоу;
- решение, которое будет дорого откатить (например, изменение state machine задачи);
- изменение `acceptance_policy`, `review_state`, `task_events`-схемы.

**Что делаем:**
- `Plan` — обязательно;
- Decision Record — обязательно (см. `decision_record.md`);
- если AI — пройти чеклист из `ai_oversight_policy.md`;
- проверить, не нарушаем ли «лоуэр-слой» из `theoretical_foundation.md` (богатство `task_events`, отделённость `review_state` от `state` и т.п.).

---

## Маппинг на типичные задачи WFM

| Задача | Режим |
|---|---|
| Поправить копирайт, цвет кнопки, иконку | FAST |
| Починить мелкий баг в форматировании времени | FAST |
| Добавить новый экран по существующей модели задач | STANDARD |
| Новое аналитическое событие на iOS+Android | STANDARD |
| Добавить фильтр в `GET /tasks/list` | STANDARD |
| Добавить колонку в `task` или `task_events` | FULL |
| Новый межсервисный эндпоинт `/internal/*` | FULL |
| Auto-приёмка фото через AI-скоринг | FULL |
| Изменение state machine задачи | FULL |
| Новая привилегия работника | FULL |
| Перенос фичи в отдельный SPM/Gradle модуль | FULL |

---

## Anti-patterns

- **Запуск `Plan` на правку текста.** Это шум, который обесценивает Plan для реальных архитектурных задач.
- **STANDARD на миграцию БД.** Любая колонка — FULL, даже nullable.
- **FAST на «маленькое» изменение `task_events`.** Срезанное поле невосстановимо (см. `theoretical_foundation.md`, §S-шаг EIM).
- **FULL на копию существующего паттерна.** Если уже есть Decision Record на «как делаем X», новые экземпляры X — STANDARD.
