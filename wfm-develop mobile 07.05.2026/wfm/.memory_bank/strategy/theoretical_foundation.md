# Теоретический фундамент архитектурных решений

Короткая выжимка теории из ВКР, **связанная с тем, как сейчас собираются данные в WFM**. Не план реализации, не академический трактат — справочник «почему текущая модель именно такая».

**Связанные документы:**
- `.memory_bank/strategy/ai_oversight_policy.md` — политика расширения AI-автономии
- `.memory_bank/analytics/dt_metric.md` — Δt как метрика дашборда
- `.memory_bank/analytics/task_events_backend.md` — аудит-лог `task_events`
- `.memory_bank/domain/task_model.md` — модель задач

---

## Зачем этот документ

ВКР проекта формализует ценность системы через две теоретических рамки: модель **EIM** (Execution Intelligence Model) и метрику **Δt** (длительность управленческого цикла). Большинство решений в текущем продукте интуитивно с ними согласуются, но связь не была явной. Без явной фиксации эта согласованность хрупкая: при следующем рефакторинге легко срезать поле, которое выглядит «ненужным», но на самом деле — носитель сигнала для будущих слоёв аналитики.

---

## Модель EIM (S→I→D→T→E→F→L)

Цикл из семи шагов:

| Шаг | Расшифровка | Реализация в WFM |
|---|---|---|
| **S** Signal | Захват сигнала о ситуации (событие, отчёт, отклонение) | ✅ `task_events`, `report_text`, `report_image_url` |
| **I** Interpretation | Машинная интерпретация сигнала | ❌ Не реализовано |
| **D** Decision | Решение (приоритезация, маршрутизация) | ❌ Не реализовано |
| **T** Task | Постановка задачи исполнителю | ✅ `POST /tasks`, LAMA-импорт |
| **E** Execution | Исполнение задачи | ✅ `start/pause/resume/complete` + state machine |
| **F** Feedback | Контроль качества | ✅ `review_state` (ACCEPTED / REJECTED) + `review_comment` |
| **L** Learning | Институциональная память (паттерны успеха/провала) | ❌ Не реализовано |

**Ключевая зависимость:** качество будущих **I** и **L** определяется тем, насколько богат текущий **S**. Историю событий нельзя обогатить задним числом — поэтому контекстные поля в `task_events` и связанных таблицах должны записываться сейчас, даже если аналитического слоя для них пока нет.

## Метрика Δt

**Определение:** Δt = `task.completed_at − task.created_at` для задач с `review_state = ACCEPTED`.

Vial (2019) определяет цифровую трансформацию через **скорость организационного ответа на сигналы среды**. Δt — это операционализация этой идеи: насколько быстро магазин закрывает обнаруженную операционную ситуацию (выкладка, переоценка, инцидент).

**Дополняет, а не заменяет ROI «снижение ФОТ»**:
- ФОТ — финансовый показатель для покупателя (понятен, но локален)
- Δt — стратегический показатель (адаптивность организации, устойчив к ценовому давлению)

Подробнее о метрике: `.memory_bank/analytics/dt_metric.md`.

## Архитектурные следствия (то, что мы УЖЕ делаем правильно)

| Решение | Чем поддерживает теорию |
|---|---|
| `task_events` пишет полную историю переходов с `actor_id`, `actor_role`, `comment`, `meta` | S-шаг EIM: богатый сигнал |
| `task` хранит `zone_id`, `work_type_id`, `shift_id`, `category_id` | контекст для будущих I/L (через JOIN с `task_events`) |
| `review_state` отделён от `state` (два измерения) | F-шаг EIM: контроль качества — это отдельная сущность, не побочный эффект исполнения |
| `acceptance_policy` (AUTO / MANUAL) | явная развилка «есть ли human-in-the-loop у этого типа работ» — основа для AI-oversight policy |
| `allow_new_operations` + модерация операций | tacit knowledge capture: формализация неявных паттернов работы через предложения с фронта |

Эта таблица — **обоснование, а не разрешение расширять**. Любая новая колонка должна оправдываться конкретной задачей, а не «теория говорит, что нужно».

---

## Что НЕ делаем «по теории»

- **Не вводим формулы EIM в код или БД.** Модель — обоснование направления, не специфический алгоритм.
- **Не строим process mining как отдельный сервис.** Текущая модель данных достаточна; аналитический слой появится тогда, когда возникнет конкретный заказчик результата.
- **Не подменяем бизнес-кейс «снижение ФОТ» на Δt.** Δt — дополнительная метрика для зрелых клиентов и внутреннего трекинга прогресса.
- **Не создаём поля под «tacit knowledge», `business_outcome`, `expected_duration` справочно.** Эмпирические оценки выводятся из `task_events` под нужную группировку.

---

## Когда возвращаться к этому документу

- Перед добавлением полей в `Task` или `task_events` — проверить, не дублирует ли это уже доступное через JOIN.
- Перед предложением «давайте подскажем менеджеру приоритет / автоматически примем фото» — пройтись по `ai_oversight_policy.md`.
- При обсуждении дашбордов и отчётов — проверить, не игнорируем ли Δt как первый класс показателя (или наоборот, не переусложняем ли его).
