# Политика человеческого надзора над AI

Документ описывает, как WFM расширяет AI-автономию **без потери человеческого контроля**. Сейчас AI в продакшене отсутствует — политика фиксируется заранее, чтобы при первом же AI-кейсе не принимать решения наспех.

**Связанные документы:**
- `.memory_bank/strategy/theoretical_foundation.md` — общий теоретический контекст
- `.memory_bank/strategy/decision_modes.md` — AI-фичи всегда FULL-режим
- `.memory_bank/strategy/decision_record.md` — ADR пишется поверх чеклиста ниже
- `.memory_bank/domain/task_model.md` — `acceptance_policy`, `review_state`
- `.memory_bank/backend/apis/api_tasks.md` — `/approve` и `/reject` endpoints

---

## Принцип

NIST AI Risk Management Framework: **«Human-in-the-loop полезен только тогда, когда заранее определено, кто именно, на каком шаге и в каком объёме осуществляет надзор».**

Каждое расширение AI-автономии (от photo verification до auto-generation задач) **требует явного решения** до мерджа: что делает AI, что остаётся за человеком, как откатить, если AI ошибся.

---

## Текущее состояние

| Контур | AI присутствует? | Кто контролирует |
|---|---|---|
| Создание задачи | Нет | MANAGER (вручную) или LAMA (импорт по плану) |
| Назначение задачи | Нет | MANAGER (`assignee_id`) |
| Выполнение задачи | Нет | WORKER (state machine) |
| Отчёт фото / текст | Нет | WORKER при `complete` |
| Приёмка отчёта | Нет | MANAGER (`approve` / `reject`) или AUTO-приёмка по политике типа работы |
| Аналитика и приоритизация | Нет | MANAGER (своё суждение) |

**Архитектурный crucial point:** поле `acceptance_policy` (AUTO / MANUAL) уже задаёт инфраструктуру развилки «нужен ли человек на финальной приёмке». Любая AI-автономия в верификации **должна продолжать этот же паттерн**, а не строить параллельный.

## Базовая матрица для будущих AI-фич

Используется как чеклист **до** реализации, не как маркетинг:

| Гипотетическая AI-фича | AI делает | Человек делает | Минимальный запрет |
|---|---|---|---|
| Photo scoring (оценка соответствия) | Считает скор и подсвечивает аномалии | MANAGER подтверждает / отклоняет в текущем UI приёмки | AI **не выставляет** `review_state = ACCEPTED` напрямую без MANAGER |
| Priority recommendation | Предлагает порядок задач для смены | MANAGER принимает или меняет порядок | AI **не пропускает** задачи без явного решения MANAGER |
| Anomaly detection (gaming, аномальная скорость) | Помечает подозрительные завершения | MANAGER рассматривает и принимает решение | Никаких автоматических санкций к работнику |
| Auto-generation задач | Генерирует черновик из шаблона | MANAGER явно утверждает шаблон → задачи создаются автоматически по нему | Без утверждённого шаблона генерация запрещена |
| Hint-обогащение | Предлагает подсказки для типов работ | MANAGER модерирует подсказки (как сейчас с операциями) | AI **не публикует** подсказку без MANAGER |

## Чеклист перед мерджем AI-фичи

1. **Что именно решает AI?** — одно предложение, без размытия.
2. **Что остаётся за человеком?** — конкретная роль (MANAGER / WORKER) и конкретное действие.
3. **Откат:** как отменить решение AI, если оно ошибочно (ручной флаг, переоткрытие задачи, override-кнопка)?
4. **Логирование:** записывается ли AI-решение в `task_events.meta` (или эквивалент), чтобы можно было пост-фактум разобрать инцидент?
5. **UI-индикация:** видит ли пользователь, что предложение пришло от AI, а не от другого человека?
6. **Метрика качества AI:** есть ли минимальный план, как мы поймём, что AI стал хуже (false positive rate, MANAGER reject rate, расхождение с ground truth)?
7. **Anti-hypothesis:** в каких условиях AI-предложение причиняет **реальный** вред (false positive с дорогим последствием для работника / магазина / клиента)? Если ответ «маловероятно» — это не ответ; нужно описать конкретный сценарий и его цену.

Без всех 7 ответов — фича не идёт в продакшн.

## Чего НЕ закладываем заранее

- **Adversarial / gaming-resistance design** как отдельную подсистему. До появления AI-скоринга это философия, а не задача.
- **Поле `business_outcome`** на задаче для «ground truth обучения». Источника заполнения нет.
- **Параллельное `review_state`-измерение для AI-вердикта.** Когда понадобится — расширим существующее (`acceptance_policy`, `task_events.meta`), а не введём новую колонку.

## Когда обновлять этот документ

- При первом же предложении ввести AI-функцию — пройтись по чеклисту, дополнить матрицу.
- При появлении регулирования (NIST AI RMF, российские нормативы по AI) — добавить ссылку на конкретный документ.
- При инциденте, где AI принял неверное решение, — добавить retro-урок.
