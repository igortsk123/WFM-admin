# ADR: Выделение svc_users в отдельный сервис

**Дата:** 2026-01-30
**Режим:** FULL
**Owner:** Pavel Karpychev
**Review trigger:** появление третьего сервиса, которому нужны данные пользователя (помимо svc_tasks и svc_notifications), либо смена SSO-провайдера / переход с Beyond Violet SSO на собственную авторизацию.

## Контекст

В svc_tasks уже жили таблицы `user_roles` и `worker_permissions`, и там же планировалось хранить расширенный профиль пользователя (должность, разряд, магазин, привязка к LAMA `external_id`). Параллельно требовалось интегрировать SSO Beyond Violet (`/sys/v1/users/{uuid}/`) для получения ФИО, фото, email — данных, которые WFM сам не ведёт. Складывать всё в svc_tasks — означало смешивать «домен задач» с «доменом идентичности» и заставлять любой будущий сервис ходить за пользователем в сервис задач.

## Решение

Вынести данные пользователей, привилегии, справочники (роли, должности, типы сотрудников) и интеграцию SSO/LAMA в отдельный сервис `svc_users` с собственной БД `wfm_users` — потому что идентичность пользователя является cross-domain ресурсом и должна жить в одном месте, независимо от домена задач.

## Альтернативы

- **Оставить пользователей в svc_tasks:** отвергнуто — любой новый сервис (notifications, shifts, будущая аналитика) был бы вынужден импортировать модели svc_tasks или обращаться к нему через HTTP за данными, не связанными с задачами.
- **Хранить только локальный профиль, всё остальное брать из SSO «налету»:** отвергнуто — SSO лежит за пределами Docker-сети, latency и доступность не контролируются; нужен локальный кэш и локальные поля (роль, должность, разряд, store_id), которых в SSO нет.
- **Вынести только справочники, сами users оставить в svc_tasks:** отвергнуто — JOIN'ы между БД невозможны, всё равно понадобился бы HTTP-клиент.

## Экономическое/архитектурное обоснование

- Чёткая граница владения: svc_users — единственный, кто знает, как пользователь выглядит «целиком» (локальные поля + SSO-кэш + LAMA `external_id`). Остальные сервисы оперируют id и при необходимости ходят через `/users/{id}` или `/users/internal/*`.
- SSO-кэш `user_sso_cache` (TTL 24 ч) с graceful degradation (отдаём stale-кэш при недоступности SSO) изолирован в одном сервисе и не дублируется по svc_tasks/svc_notifications.
- Подготовлена площадка для подключения LAMA как второго внешнего источника без переделки контрактов потребителей.

## Принятые риски

- Каждый запрос svc_tasks → svc_users добавляет HTTP-хоп в Docker-сети; компенсируется in-memory кэшем в `users_client`.
- Две БД нужно бэкапить и мигрировать раздельно; rollback миграции, затрагивающей оба сервиса, требует координации.
- При недоступности svc_users парализуются все остальные сервисы (резолв `sso_id → user_id` идёт через него) — фактически он становится критическим путём.

## Future hook

- Таблица `user_sso_cache` отделена от `users` и хранит только то, что приходит из SSO — её можно переключить на event-driven обновление (webhook от SSO) без затрагивания основной модели пользователя.
- Внутренние эндпоинты вынесены под префикс `/internal/*` без JWT (только Docker-сеть) — готова поверхность для будущих межсервисных вызовов (id-by-sso уже там, см. ADR `2026-03-09_internal_int_user_id`).
- `external_id` (LAMA) вынесен отдельным полем на User — при появлении третьего внешнего источника достаточно добавить ещё одно поле, не меняя схему.
