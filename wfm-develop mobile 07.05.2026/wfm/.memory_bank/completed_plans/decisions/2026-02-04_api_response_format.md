# ADR: Единый формат ответов WFM API

**Дата:** 2026-02-04
**Режим:** FULL
**Owner:** Pavel Karpychev
**Review trigger:** появление второго внешнего потребителя API (помимо мобильных клиентов и LAMA), либо запрос от партнёра на ответы в RFC 7807 / JSON:API.

## Контекст

Сервисы svc_tasks и svc_users возвращали ответы в произвольном виде: где-то «голый» объект, где-то FastAPI-дефолт `{"detail": "..."}`. Мобильные клиенты и LAMA уже работают с Beyond Violet API, у которого формат фиксирован. Без единой обёртки клиенты вынуждены писать парсер на каждый эндпоинт и угадывать, где искать код ошибки.

## Решение

Все WFM API отвечают в формате `{status: {code, message?}, data?}` — потому что это совместимо с уже привычной обёрткой Beyond Violet API и даёт клиентам один контракт парсинга для успехов и ошибок.

## Альтернативы

- **RFC 7807 (`application/problem+json`) только для ошибок:** отвергнуто — оставляет два разных формата (успех/ошибка), мобильные клиенты всё равно должны иметь два пути парсинга.
- **JSON:API:** отвергнуто — избыточен для closed-domain MVP, требует `included`/`relationships`, которые мы не используем.
- **FastAPI дефолт + кастомные исключения:** отвергнуто — нет единого места для `code`, `message` ошибки приходит как `detail` вперемешку с внутренними сообщениями pydantic.

## Экономическое/архитектурное обоснование

- Один shared-модуль `backend/shared/` (схемы `ApiResponse`, `StatusSchema`, exception-классы, глобальный handler) подключается ко всем сервисам через docker build context — стоимость изменения формата нулевая для нового сервиса.
- Мобильные клиенты получают одну функцию-парсер для всех endpoint'ов вместо N разрозненных.
- Совместимость с Beyond Violet API снимает когнитивную нагрузку при чтении/документировании: одинаковая структура ответов на двух соседних бэкендах.

## Принятые риски

- Любое изменение формата (например, добавление `meta` для пагинации) теперь требует одновременного обновления всех сервисов и мобильных клиентов — обёртка стала межсервисным контрактом.
- Глобальный exception handler перехватывает все необработанные исключения и оборачивает их в `status.code = INTERNAL_ERROR`, скрывая stack trace из ответа — отладка требует обращения к логам, а не к телу ответа.
- Поле `data` опционально и в ошибках отсутствует — клиенты обязаны проверять `status.code` до обращения к `data`.

## Future hook

Поле `status` оставлено как объект (а не плоская строка) — туда можно добавить `request_id`, `trace_id` или массив `details[]` без breaking change. Поле `data` намеренно может быть и объектом, и контейнером со списком (`{tasks: [...]}`) — это позволит позже добавить пагинацию через `data.meta` без редизайна.
