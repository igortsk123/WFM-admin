# ADR: ESP32-устройство — JC4827W543N как целевая платформа и single-screen dashboard как UI

**Дата:** 2026-04-27
**Режим:** FULL
**Owner:** Pavel Karpychev
**Review trigger:** появление целевого Waveshare AMOLED 480×480 в продакшене (актуальность портирования драйверов), отказ JC4827W543N с рынка/из поставок, либо запрос на интерактивность устройства (тач, кнопки) — тогда single-screen read-only концепция требует пересмотра.

## Контекст

Прошивка физического ESP32-монитора инфраструктуры исторически разрабатывалась на простом тест-стенде (ST7789 240×240, SPI). По мере роста объёма данных стало очевидно: квадратик 240×240 не вмещает даже метрики хоста + критичные контейнеры + счётчик ошибок одновременно — приходилось ротировать три экрана раз в 5 секунд, что в роли «фонового монитора краем глаза» работало плохо. Параллельно встал вопрос о платформе для следующей итерации, на которой можно собирать реальное устройство для офиса.

## Решение

Целевой платформой выбран **JC4827W543N** (4.3" AMOLED 480×272, NV3041A по QSPI, ESP32-S3, без тача), а UI переведён на **single-screen dashboard с автообновлением через WebSocket каждые 30 с**, без меню и навигации — потому что устройство читает один WS-канал и должно показывать всё критичное «одним взглядом», а не требовать взаимодействия.

Холст рендера остаётся 480×480 (как требует будущий целевой Waveshare AMOLED), на JC4827W543N viewport 480×272 циклически page-switch'ится между верхней и нижней половиной холста каждые 5 с. Никаких отдельных раскладок под temp vs target — один `screen_dashboard()` едет на оба устройства.

## Альтернативы

**По платформе:**
- **Остаться на ST7789 240×240 SPI:** отвергнуто — физически не вмещает dashboard, экран мутный, нерабочий для роли офисного монитора.
- **Waveshare 480×480 AMOLED как тест-устройство:** отвергнуто на этом шаге — на момент решения недоступен в желаемом форм-факторе/сроках; JC4827W543N выступает как мост, на котором можно отладить рендер-конвейер 1:1 без даунскейла.
- **Любая другая ESP32-плата с IPS:** отвергнуто — IPS даёт хуже контраст для тёмной темы дашборда, AMOLED показывает чёрный «честно».

**По UI:**
- **Многоэкранная ротация (как было: METRICS / CONTAINERS / ERRORS):** отвергнуто — за 5 секунд на нужный экран можно не успеть посмотреть, постоянное движение раздражает в фоне.
- **Меню/кнопки/тач для выбора экрана:** отвергнуто — устройство read-only, никаких user-actions не предполагается; добавление input'а ломает «glance-only» концепцию и требует тач-версии платы.
- **Vertical pan с easeInOut-анимацией между двумя половинами дашборда:** отвергнуто на MVP — жёсткий page-switch проще и не отвлекает; анимацию можно добавить отдельной задачей, если резкость станет проблемой.
- **Отдельная landscape-раскладка 480×272 для JC4827W543N:** отвергнуто — двойная разметка ломает совместимость с целевым 480×480, рендер пришлось бы переписывать дважды.

## Экономическое/архитектурное обоснование

- AMOLED 480×272 даёт читаемый dashboard с компактными примитивами (`MiniBar`, `MiniServiceRow`, `errors-pill`) без даунскейла — pixel-perfect 1:1.
- Холст 480×480 как «дизайн-система устройства» означает, что портирование на целевой Waveshare сводится к замене драйвера дисплея — раскладка и рендер-код переезжают как есть.
- Source of truth для контента — JSON от `svc_monitoring` (`state` каждые 30 с + `alert` мгновенно). Контейнеры рендерятся в порядке прихода с сервера (фикс `set → tuple` в `CRITICAL_CONTAINERS`) — устройство тупо отображает то, что прислали, без локальной логики сортировки.
- Read-only концепция убирает целый класс задач: нет настроек, нет состояний UI, нет reconciliation между локальным и серверным состоянием.

## Принятые риски

- **Зависимость от конкретной модели:** JC4827W543N — китайская плата с NV3041A; снятие с производства / смена драйвера потребует портирования (драйверы и init-последовательности нетривиальны).
- **Нестандартный QSPI-стек:** библиотека `Arduino_GFX` для NV3041A менее развита, чем TFT_eSPI/LVGL для классических SPI-дисплеев — нет AA на дугах, шрифты — встроенные Adafruit GFX (не дизайн-точные), drawArc заменён на самопальный fillArc + конвертация углов.
- **Page-switch без анимации:** жёсткое переключение каждые 5 с может раздражать в периферийном зрении; не выяснено, переживёт ли это многочасовая работа в офисе.
- **PSRAM bound:** холст 480×480×2 = 450 KB на плате с 8 MB OPI PSRAM — запас есть, но любое расширение (двойная буферизация для плавного pan) упрётся в память.
- **Нет тача:** если в будущем понадобится показать детали ошибки/контейнера по клику — придётся менять плату (версия N — без тача), либо параллельно держать второй прототип с тачем.
- **Отсутствие primary-выделения контейнеров:** все контейнеры рендерятся одинаково, постгрес визуально не выделен — оператор должен помнить иерархию критичности сам.

## Future hook

- Холст 480×480 = **переезд на Waveshare AMOLED 480×480 без перерисовки**, только смена драйвера и удаление page-switch (всё помещается). Это явно зашитая инвестиция в следующий слой.
- Pan-логика в `display.cpp` изолирована (`push_frame_native(y_offset)`) — добавление easeInOut между крайними положениями = одна функция, без правок `screen_dashboard()`.
- `screen_alert` оставлен как полноэкранный оверлей с контентом в зоне y=0..272 — переезд на 480×480 требует только расширения позиционирования, логика и контент сохраняются.
- Структура контейнеров на устройстве — массив `{name, status}` без именованных полей (`state.h`) — добавление пятого/шестого критичного сервиса на сервере не требует ничего на устройстве, только увеличить `MAX_CONTAINERS` если перерастём.
