# ADR: Двойная push-доставка через Firebase (Google) и HMS (Huawei)

**Дата:** 2026-04-01
**Режим:** FULL
**Owner:** Pavel Karpychev
**Review trigger:** Huawei разрешит/потребует FCM на своих устройствах, либо появится третий push-провайдер (RuStore Push, Apple-эквивалент для Android), либо доля HMS-устройств в пользовательской базе упадёт ниже порога окупаемости поддержки.

## Контекст

Система уведомлений построена на Firebase Cloud Messaging. Часть целевых пользователей — сотрудники с Huawei-устройствами без Google Play Services, на которые FCM-токены не выдаются и push'и не доставляются. Без альтернативного канала эти пользователи остаются без real-time уведомлений о задачах.

## Решение

Push'и доставляются параллельно через FCM (для GMS-устройств и iOS) и HMS Push Kit (для Huawei) — потому что это единственный способ покрыть Huawei без Google-сервисов, не отказываясь от уже работающего FCM-стека для основной массы пользователей.

Тип токена определяется на клиенте по доступности сервисов и помечается полем `token_type` (`fcm` / `hms`); бэкенд хранит его в `device_tokens` и маршрутизирует доставку по типу. Android собирается в двух flavor'ах (`gms` / `hms`) — каждый включает только свой SDK, чтобы не раздувать APK.

## Альтернативы

- **Только FCM:** отвергнуто — на Huawei-устройствах без GMS push'и просто не работают, real-time канал отсутствует.
- **Только HMS для всех Android:** отвергнуто — HMS Push Kit на устройствах с GMS работает менее надёжно и теряет совместимость с Firebase Analytics/Crashlytics.
- **Web Push / только WebSocket:** отвергнуто — WS не работает при закрытом приложении и в фоне с агрессивным энергосбережением (особенно на Huawei), а Web Push на нативном Android требует отдельной обвязки.
- **Сторонний агрегатор (OneSignal и т.п.):** отвергнут — добавляет внешнюю зависимость, отдаёт пользовательские токены наружу, не снимает необходимость иметь HMS App ID и подписанные конфиги.

## Экономическое/архитектурное обоснование

- Абстракция `PushService` на Android (интерфейс + flavor-specific реализация) изолирует SDK-специфику — общий код не знает, какой провайдер активен.
- Поле `token_type` в `device_tokens` + маршрутизация в `delivery_engine.py` — единственная развилка на бэкенде; добавление третьего провайдера сводится к новому значению enum + новой ветке доставки.
- Backward compat для старых клиентов: `platform: "ANDROID"` без `token_type` принимается и трактуется как `AND` + `fcm` — миграция пользовательских токенов не требуется.
- Flavor dimension `brand` уже использовался для дуальной сборки приложений — добавление dimension'а для push-провайдера переиспользует ту же gradle-инфраструктуру.

## Принятые риски

- **Дублирование push-инфраструктуры:** два набора credentials (FCM service account + HMS App ID/Client Secret), два места для отзыва токенов, два канала ошибок доставки.
- **Удвоение тестирования:** каждое изменение в push-логике требует прогона на устройствах обеих экосистем; HMS-устройство — отдельная физическая железка, недоступная в большинстве облачных farm'ов.
- **Удвоение Android-сборок:** CI собирает четыре варианта (gms × hms × debug × release × два бренда = 8 APK), время сборки и хранилище артефактов растут пропорционально.
- **Расхождение API между FCM и HMS:** форматы payload, коды ошибок невалидных токенов, лимиты на размер — разные. Маппинг внутри `hms_client.py` придётся поддерживать вручную при изменениях SDK.
- **OAuth-токен HMS истекает:** требует кэширования и retry-логики (`80200003` → refresh) — дополнительный путь отказа, которого нет в FCM.

## Future hook

`token_type` в `device_tokens` — открытый enum строка, не bit field; добавление RuStore Push (`rustore`) или любого другого провайдера = миграция на enum с новым значением + новая ветка в `delivery_engine`, без изменения схемы. Платформенный код `platform` (`AND`/`HUA`/`IOS`) разведён с `token_type` намеренно — это позволит, например, выдавать HMS-токены на iOS-альтернативах в Китае без переделки модели.
