# ADR: CI/CD pipeline через собственный Docker Registry + Watchtower

**Дата:** 2026-02-08
**Режим:** FULL
**Owner:** Pavel Karpychev
**Review trigger:** появление третьего окружения (помимо DEV/PROD), необходимость blue-green/canary деплоя, либо рост числа сервисов > 8 — тогда polling Watchtower и плоская топология станут узким местом.

## Контекст

Backend WFM состоит из нескольких FastAPI-сервисов, развёртываемых на двух серверах (DEV и PROD) через docker-compose. До этого момента деплой выполнялся вручную через SSH + `docker compose pull/up` — каждый релиз требовал ручных действий, а ветка `develop` и теги `v*.*.*` никак не были связаны с тем, что реально крутится на серверах.

## Решение

CI/CD строится по схеме «GitVerse CI собирает образ → пушит в собственный Docker Registry на `registry.beyondviolet.com` → Watchtower на сервере по polling'у подтягивает новый образ» — потому что это даёт полный автодеплой без SSH-доступа из CI и без зависимости от внешних managed-решений.

## Альтернативы

- **Ручной SSH-деплой (как было):** отвергнут — каждый релиз требует человека, нет аудита «что сейчас на проде», легко забыть `docker compose pull`.
- **Kubernetes (k3s/k8s):** отвергнут — overkill для двух серверов и MVP-нагрузки, требует выделенного времени на обслуживание control-plane и сетевого слоя.
- **GitHub Container Registry / Docker Hub:** отвергнуто — приватные образы у внешнего registry привязывают к стороннему вендору и его лимитам/ценам; собственный registry уже стоит для других проектов Beyond Violet.
- **CI с push по SSH (`docker compose pull && up` из workflow):** отвергнут — требует хранить SSH-ключи в секретах CI, даёт CI полный root на проде, ломается при любых сетевых перебоях между CI и сервером.

## Экономическое/архитектурное обоснование

- Одно изменение в коде → один git push → автоматический деплой на нужное окружение (`develop` → DEV `:dev` тэг, `v*.*.*` → PROD `:latest` + версионный тэг).
- Watchtower работает с любым docker-compose стеком без дополнительной интеграции — добавление нового сервиса требует только описания в `compose.yml`, никаких доработок pipeline.
- Собственный registry уже существует как инфраструктура Beyond Violet — нулевая дополнительная стоимость; политика хранения и доступа — в наших руках.
- Разделение DEV/PROD по тэгам (`:dev` vs `:latest`) изолирует окружения на уровне реестра — невозможно случайно задеплоить dev-сборку на прод.

## Принятые риски

- **Latency деплоя:** Watchtower polling'ит реестр с фиксированным интервалом — между push'ом образа и фактическим запуском на сервере может пройти до нескольких минут. Hot-fix требует ручного триггера или ожидания.
- **Watchtower как single point of failure:** если демон Watchtower упал/завис — деплои тихо встают, пока никто не заметит. Нет встроенного healthcheck'а самого Watchtower'а.
- **Собственный registry требует обслуживания:** диск под слои, garbage collection старых образов, бэкапы, обновление версии registry — всё на нашей стороне.
- **Нет отката одной командой:** rollback = пересобрать предыдущий тэг или вручную `docker tag` + restart. CI не хранит карту «версия → коммит».
- **Секреты в `.env` на серверах:** CI не управляет секретами окружения — изменения требуют ручного редактирования `.env` на сервере и перезапуска.

## Future hook

Тэги в реестре уже разделены на `:dev` / `:latest` / `:vX.Y.Z` — добавление третьего окружения (staging) или canary-развёртывания сводится к новому тэгу и второму инстансу Watchtower с фильтром по этому тэгу, без переделки pipeline. Версионные тэги `vX.Y.Z` сохраняются — это база для будущего ручного rollback'а через `docker compose pull <tag>`.
