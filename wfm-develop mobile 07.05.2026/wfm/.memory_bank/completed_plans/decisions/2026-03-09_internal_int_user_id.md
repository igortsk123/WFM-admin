# ADR: Internal integer user_id вместо SSO UUID

**Дата:** 2026-03-09
**Режим:** FULL
**Owner:** Pavel Karpychev
**Review trigger:** смена SSO-провайдера, появление federated identity (несколько SSO одновременно), либо потребность синхронизировать пользователей через event bus с external_id как первичным ключом.

## Контекст

JWT от Beyond Violet SSO содержит `u: <uuid>`, и до этой точки UUID был основным `id` пользователя сразу во всех сервисах: `users.id`, `tasks.creator_id`, `tasks.assignee_id`, `task_events.actor_id`, `permissions.user_id`. Это означало, что SSO-идентификатор внешней системы был **первичным ключом** во всех внутренних таблицах и индексах. Любой инцидент со стороны SSO (сброс UUID, миграция, замена провайдера) ломал бы внутреннюю целостность WFM. Параллельно появлялся LAMA `external_id` как ещё один внешний идентификатор, и стало очевидно, что внешние ID не должны быть нашими первичными ключами.

## Решение

Внутренние сервисы оперируют auto-increment integer `user_id` (собственный PK), а SSO UUID хранится отдельным полем `users.sso_id` (unique, not null) только для маппинга при JWT-аутентификации — потому что внешние идентификаторы не должны определять внутреннюю схему БД и индексы.

## Альтернативы

- **Оставить UUID как PK:** отвергнуто — внешний идентификатор привязывается к нашим индексам, foreign keys и API-контрактам; смена SSO становится катастрофическим событием.
- **Использовать UUIDv7 как PK (sortable):** отвергнуто — всё равно остаётся коллизия с тем, что значение приходит извне; меньше места не экономит и никак не упрощает миграцию.
- **Хранить оба и принимать оба в API:** отвергнуто — раздваивает контракт, заставляет каждый сервис помнить, какой ID он сейчас держит.
- **Хешировать UUID → int:** отвергнуто — теряется обратная связь, нет защиты от коллизий, дополнительная сложность без выгод над auto-increment.

## Экономическое/архитектурное обоснование

- Внутренние foreign keys, индексы и фильтры работают на 4-байтных integer'ах вместо 16-байтных UUID — мелкая, но измеримая экономия на JOIN'ах и индексах для горячих запросов (`tasks.assignee_id`, `task_events.actor_id`).
- Резолв `sso_id → user_id` происходит один раз при аутентификации (`shared/auth.validate_token_and_get_sso_id` + DB lookup в svc_users); внутренние сервисы получают уже готовый int и не знают про SSO.
- Появилась чистая поверхность для второго внешнего ID: LAMA `external_id` лежит на той же таблице рядом с `sso_id` без конфликтов и без переписывания межсервисных контрактов.
- Подготовлена база для миграции SSO-провайдера: достаточно перевыпустить `sso_id` в `users` — все остальные таблицы и сервисы не затрагиваются.

## Принятые риски

- **Дополнительный DB lookup на каждый запрос:** компенсирован in-memory кэшем в `users_client` (svc_tasks → svc_users) с TTL и invalidation на logout.
- **Breaking change для всех клиентов:** мобильные iOS/Android прошли одновременную миграцию (`User.id: String → Int`, добавлен `ssoId: String`); web-клиента ещё нет, поэтому окно совместимости не нужно было.
- **Скрипт миграции данных в svc_tasks:** требовалось пройти по всем UUID в `tasks.creator_id`, `tasks.assignee_id`, `task_events.actor_id` и заменить их на int через `/users/internal/id-by-sso`. Скрипт `data_migration.py` выполнялся вручную — на DEV прошёл, для PROD требовался отдельный maintenance window.
- **Permission.id оставлен UUID:** сознательно, чтобы не плодить миграции там, где нет user identity (PK самой записи привилегии).

## Future hook

- `users.sso_id` — отдельный nullable-готовый слой; при миграции на новый SSO достаточно перевыпустить `sso_id` без затрагивания остальной схемы.
- Внутренний эндпоинт `/users/internal/id-by-sso?sso_id=X → {id: int}` — стабильный контракт для будущих потребителей (например, импортёры, миграторы, BI-инструменты, синхронизация с LAMA).
- `external_id` (LAMA) лежит рядом с `sso_id` — паттерн «множественные внешние ID при одном внутреннем» уже зафиксирован, добавление третьего источника тривиально.
- `task_events.actor_id` — int, что облегчает будущий аналитический слой на событиях (см. `.memory_bank/strategy/theoretical_foundation.md`): join'ы и агрегации по int-ключам быстрее и компактнее.
