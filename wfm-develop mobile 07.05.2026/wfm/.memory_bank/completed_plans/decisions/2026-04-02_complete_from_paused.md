# ADR: Разрешить переход PAUSED → COMPLETED напрямую

**Дата:** 2026-04-02
**Режим:** FULL
**Owner:** Pavel Karpychev
**Review trigger:** при добавлении нового состояния в task state machine; при изменении правила «одна активная задача на работника» (если активной задачей будет считаться и PAUSED).

## Контекст

State machine задач требовала, чтобы перед `complete` задача обязательно была переведена в `IN_PROGRESS` — то есть из паузы работник сначала делал `resume`, потом `complete`. На UX это выглядело как лишний шаг: работник, решивший закрыть задачу из паузы, был вынужден сначала «возобновить» её на пару секунд. Переход добавлял лишнее событие в `task_events` без полезной информации и провоцировал ошибки клиентов (попытка `complete` из PAUSED → 409).

## Решение

Разрешаем прямой переход `PAUSED → COMPLETED` через тот же endpoint `/complete`, потому что промежуточный `resume` не нёс семантической нагрузки и только усложнял клиентский код и аудит-лог.

## Альтернативы

- **A: Оставить старое поведение (требовать resume)** — отвергнуто: лишний клиентский цикл, лишнее событие `RESUME` в `task_events`, выглядит непредсказуемо для пользователя.
- **B: Сделать `complete` доступным из любого состояния (включая NEW)** — отвергнуто: завершение задачи без её начала ломает аналитику длительности (`completed_at - started_at`), теряется честный сигнал о факте исполнения; для NEW корректное действие — отмена/удаление, не завершение.
- **C: Ввести явный комбинированный endpoint `/resume-and-complete`** — отвергнуто: это API ради процесса, а не ради состояния; то же самое можно решить расширением state machine.

## Экономическое/архитектурное обоснование

Микро-улучшение UX, которое тем не менее формализуется как ADR (FULL-режим), потому что **любое изменение state machine** требует фиксации — иначе через год невозможно понять, почему переход существует. Изменение точечное, обратно совместимо (старый путь PAUSED → IN_PROGRESS → COMPLETED продолжает работать).

## Принятые риски

- В `task_events` исчезает событие `RESUME` для случаев завершения из паузы — аналитика длительности «активного» времени (`IN_PROGRESS`) недосчитывает короткие resume-сессии. Считаем приемлемым: реальной активной работы перед таким `complete` не было.
- Клиенты, ранее завязанные на ожидание перехода в IN_PROGRESS перед `complete`, нужно проверить (в текущих iOS/Android — кнопка «Завершить» уже была видна в PAUSED, ломаться нечему).

## Future hook

Если будет введено состояние с временной отметкой «фактически работал N минут», `PAUSED → COMPLETED` нужно будет учитывать как «N=0 для последней сессии». Текущее решение этому не мешает: `task_events` сохраняет полную историю, можно посчитать активное время постфактум.
