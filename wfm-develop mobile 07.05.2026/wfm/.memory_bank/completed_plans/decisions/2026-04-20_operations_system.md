# ADR: Модерируемая система операций с предложениями от работников

**Дата:** 2026-04-20
**Режим:** FULL
**Owner:** Pavel Karpychev
**Review trigger:** при росте PENDING-операций без модерации > 50 одновременно; при появлении второго клиента кроме LAMA, у которого свой справочник операций; при предложении автоматизировать модерацию через AI.

## Контекст

Существовавший справочник `operations` был статичным: набор шагов выполнения с hint-полями, привязанный к парам (work_type, zone). Не было ни связи операций с конкретной задачей (что именно выполнил работник), ни механизма расширения справочника снизу — менеджер должен был руками добавлять каждую новую операцию, что плохо работает для разнородной сети магазинов с разной спецификой.

## Решение

Делаем операции отдельной модерируемой сущностью с собственным `review_state` (PENDING / ACCEPTED / REJECTED) и разрешаем работникам предлагать новые операции при `/complete` (если у `work_type` стоит флаг `allow_new_operations`) — потому что это превращает операционную базу в совместный артефакт менеджера и работников и даёт «tacit knowledge capture» без блокировки исполнения задач.

Привязка выполненных операций к задаче — отдельная таблица `task_completed_operations`; новые операции от работников создаются с `review_state=PENDING` и появляются в общем списке для пары (work_type, zone) сразу, но помечены как непроверенные.

## Альтернативы

- **A: Только менеджер заводит операции, работник выбирает из готового списка** — отвергнуто: не масштабируется на 300+ магазинов с локальной спецификой, менеджеры не знают, какие шаги фактически делает работник.
- **B: Свободный текст в отчёте без структурированных операций** — отвергнуто: невозможно строить аналитику по типам выполняемых работ, нельзя переиспользовать опыт магазина.
- **C: Делать operations.review_state = ACCEPTED по умолчанию для всех предложенных** — отвергнуто: справочник быстро замусорится синонимами и опечатками («Выкладка», «выкладка товара», «Расстановка»).

## Экономическое/архитектурное обоснование

Реиспользование `review_state` той же семантики, что у задач (см. ADR `2026-02-27_task_review_state_and_audit_log`), снижает когнитивную нагрузку: одна модель приёмки на разные сущности. Флаг `allow_new_operations` на уровне `work_type` — точечный switch для контролируемого ввода фичи (включаем для тех типов работ, где локальная вариативность ожидаема). Хранение `operation_ids` и `new_operation_ids` в `task_events.meta` сохраняет полный контекст завершения задачи в существующем аудит-логе без новой таблицы событий.

## Принятые риски

- **Накопление мусорных PENDING-операций**, которые никто не модерирует — основной риск. Без процесса разбора PENDING'ов справочник распухает дубликатами и становится менее удобным, чем был.
- **Дубликаты по смыслу, разные по написанию** («Выкладка», «выкладка товара») — модерация менеджером — единственный фильтр; нет дедупликации по нормализованному имени.
- **Невозможность удалить REJECTED-операцию**: задача в истории может на неё ссылаться, поэтому soft-delete через статус. Размер таблицы будет монотонно расти.
- **Ответ `GET /tasks/{id}` стал тяжелее** из-за `_populate_task_operations`: для задач с большим work_type список может быть длинным.

## Future hook

`operations.review_state` повторяет ту же модель, что и `tasks.review_state` — при появлении AI-модерации (например, классификатор «дубликат / новая операция») место подключения уже есть: новый `actor_role=ai` в действии approve/reject, без миграций. Структура `task_completed_operations` готова к будущей метрике «средняя доля выполненных операций по типу работ» — она строится JOIN'ом без новых полей.
