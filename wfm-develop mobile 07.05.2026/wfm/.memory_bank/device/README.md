# Device Integration — Физические мониторы

Раздел описывает интеграцию физического ESP32-устройства с сервисом `svc_monitoring` для получения real-time уведомлений о состоянии инфраструктуры WFM.

**Связанные документы:**
- `.memory_bank/backend/services/svc_monitoring.md` — серверная сторона (источник snapshot'а и push-канала)
- `.memory_bank/backend/apis/api_monitoring.md` — контракты эндпоинтов
- `.memory_bank/analytics/server_events.md` — реестр событий, которые сервис параллельно шлёт в Semetrics (исторический ряд)

Прошивка живёт в `backend/svc_monitoring/firmware/` (PlatformIO + Arduino + Arduino_GFX).

---

## Зачем это нужно

Обычные дашборды требуют, чтобы человек сам открыл браузер и проверил состояние системы. Физическое устройство решает обратную задачу: **сервер сам тебя находит**, где бы ты ни был.

Два сценария использования:

### 1. Pocket mode — носить с собой
Устройство лежит в кармане. Если сервер недоступен или что-то упало — оно подаёт сигнал. Ты знаешь о проблеме раньше, чем пользователи напишут в поддержку.

- Множество сохранённых WiFi-сетей (дом, офис, ...) с приоритетами
- Каждые 5 минут сканирует окружение и пробует подключиться к открытым точкам
- Если WebSocket недоступен больше 30 минут — звуковой сигнал

### 2. Desk mode — наблюдать после деплоя
Устройство стоит на столе. После выкатки или перезапуска контейнеров смотришь одним взглядом: всё зелёное — можно работать дальше.

- Загрузка CPU, RAM, диска в реальном времени
- Состояние критических контейнеров (healthy / stopped / unhealthy)
- Счётчик 500-х ошибок за последний час

## Общие принципы

- **Pull = плохо, Push = хорошо.** Устройство не опрашивает сервер — сервер сам пушит состояние через WebSocket.
- **Одно соединение, полное состояние.** При подключении устройство сразу получает текущий snapshot, не ждёт следующего цикла.
- **Критические события — немедленно.** Падение контейнера или пик 500-х не ждут 30-секундного таймера.
- **Прошивка не знает о бизнес-логике.** Она рендерит то, что пришло по WebSocket. Вся аналитическая логика — на сервере.
- **Экран маленький — текст бесполезен.** Только цвет, форма, анимация. Смысл должен быть понятен с расстояния метра.

## Источники данных в WFM

Критически важные данные собирает `svc_monitoring` (см. описание сервиса):
- метрики хоста через `psutil` → CPU %, RAM %, Disk %
- Docker events через unix-сокет → состояние `postgres`, `svc_tasks`, `svc_users`, `svc_notifications`
- HTTP 500+ от других сервисов через `POST /internal/errors` → счётчик `errors_1h` и алерт-спайки

Параллельно те же события (`server_metrics`, `docker_container_event`, `api_error`) уходят в Semetrics — это исторический ряд для анализа стабильности, см. `.memory_bank/analytics/server_events.md`.

## Архитектура интеграции (кратко)

```
psutil + docker.sock           svc_tasks/svc_users/svc_notifications
        ↓                                       ↓ POST /internal/errors
        └─────────► svc_monitoring (in-memory state) ◄────────┘
                            │
                            ▼
                    WSS /api/events/ws
                            │
                            ▼
                       ESP32 device
                       ├─ TFT/AMOLED display
                       └─ buzzer alert
```

Устройство ходит **напрямую** к `svc_monitoring`, без промежуточного хранения событий и polling'а — сервис уже владеет данными в источнике (psutil + docker.sock + ошибки от WFM-сервисов).
