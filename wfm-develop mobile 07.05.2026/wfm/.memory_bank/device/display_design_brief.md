# Design Brief: ESP32 Server Monitor — Display UI

## Статус тест-устройств

> **Целевое устройство** ещё не получено. Разработка ведётся на временных дисплеях.
> Все временные устройства используют **квадратную область** — приближение к форм-фактору цели.

| # | Устройство | Статус | Примечание |
|---|---|---|---|
| Временное 1 | ST7789 240×240 | Заменено | Первая тест-платформа |
| **Временное 2** | **Guition JC4827W543N 480×272** | **Активно** | 272×272 квадрат в центре |
| Целевое | Waveshare ESP32-S3-Touch-AMOLED-2.16 | Ожидается | 480×480, CO5300, QSPI |

## Контекст устройства (временное #2: JC4827W543N)

| Параметр | Значение |
|---|---|
| MCU | ESP32-S3-WROOM-1 / N4R8 |
| PSRAM | 8 MB OPI |
| Flash | 4 MB |
| Физический дисплей | NV3041A, TFT IPS 480×272 px, QSPI |
| **Рабочая область** | **272×272 px** (квадрат, x_offset=104, вся высота) |
| Библиотека | Arduino_GFX + Arduino_Canvas (double buffer) |
| Управление | Нет (нет тача, версия N) |

### Про разрешение и память

Дизайн разрабатывается под **480×480 px** — это разрешение холста (Canvas).

**Стратегия вывода на JC4827W543N (480×272):** pixel-perfect 1:1, без downscale.
Холст 480×480 рендерится целиком, на физический экран попадает viewport 480×272,
который циклически переключается между верхней (`y=0`) и нижней (`y=208`) половиной
холста каждые 5 секунд (`screen_dashboard()` в `backend/svc_monitoring/firmware/src/display.cpp`).
Хедер dashboard рендерится дважды (на `y=0` и `y=208`), чтобы статус коннекта
был виден в обеих половинах.

При выводе на **целевое устройство** 480×480 AMOLED — 1:1 без vertical-pan, всё
помещается за один кадр.

Расчёт памяти холста: `480 × 480 × 2 байта = 460 800 байт ≈ 450 KB` — вписывается
в 8 MB PSRAM (< 6% использования).

## Экраны и данные

### Screen 0 — DASHBOARD (актуальный)

**Единый экран** — заменил ротацию из 3 (METRICS / CONTAINERS / ERRORS).
Референс: `.memory_bank/device/design_handoff/monitor-screens.html` → компонент `ScreenDashboard`.

**Структура (canvas 480×480):**
- Header: `WFM` слева, справа: `[NO WIFI Xm Ys]` (если связи нет) → wifi-icon → `● LIVE/STALE`
- Divider
- METRICS: лейбл + 3 mini-bar (`CPU` / `MEM` / `DISK`) с цветом по зонам
- Divider
- CONTAINERS: лейбл + 4 mini-card в порядке прихода с сервера (накладывается на ERRORS pill справа)
- ERRORS / 1H: лейбл + крупная цветная пилюля с числом

**Pan-режим:** на JC4827W543N viewport 480×272 переключается top↔bottom каждые 5 сек.
Хедер дублируется на обеих позициях (canvas y=0 и y=208) для постоянной видимости статуса.

Реализация: `screen_dashboard()` в `backend/svc_monitoring/firmware/src/display.cpp`.

### Исторические экраны (1–3) — раскладки секций dashboard

> Эти раскладки больше не выводятся отдельно, но описывают визуальные правила
> для соответствующих секций единого dashboard.

### Screen 1 — METRICS

**Данные:** `cpu: float`, `mem: float`, `disk: float` (0–100 или -1 если нет данных)

**Текущие проблемы:**
- Три дуговых gauge в горизонтальную строку (r=36px @ 240px) — тесно, мелко
- Нижняя половина экрана пустая, верхняя — одна мелкая надпись "SERVER"
- С расстояния 1–1.5 м ничего не читается

**Что хочется:**
- Вертикальная информационная иерархия: главная загруженная метрика крупно сверху, остальные меньше
- Возможен вариант 2+1 (две маленьких снизу, одна крупная сверху) или список с прогресс-барами
- Индикатор freshness данных (timestamp или точка "live")

---

### Screen 2 — CONTAINERS

**Данные:** 4 сервиса с именами и статусами:

| Имя | Значимость |
|---|---|
| `postgres` | критичный (зависимость для всех) |
| `svc_tasks` | основной |
| `svc_users` | основной |
| `svc_notifications` | вспомогательный |

Статусы: `Healthy` / `Starting` / `Unhealthy` / `Stopped` / `Unknown`

**Текущие проблемы:**
- 4 закрашенных кружка — читается только цвет, имя усечено
- Нет иерархии важности, нет словесного статуса

**Что хочется:**
- Карточный layout или список (4 строки) с цветовым индикатором слева
- `postgres` визуально выделен (позиция, размер, рамка)
- Имя + словесный статус (не только цвет)

---

### Screen 3 — ERRORS

**Данные:** `errors_1h: int` — количество ошибок за последний час

**Текущие проблемы:**
- Гигантское число по центру без контекста нормы

**Что хочется:**
- Крупное число + шкала нормы (0 = зелёный, 1–4 = жёлтый, 5+ = красный)
- Возможно дуга или mini-chart как контекст (хотя бы визуальный диапазон)
- Подпись "за 1 час" читаемо

---

### Screen 4 — NO CONNECTION (НЕ ИСПОЛЬЗУЕТСЯ на устройстве)

> Состояние коннективности теперь живёт **в хедере dashboard** (badge `NO WIFI` /
> `NO SERVER` + длительность). Полноэкранный no-connection-экран удалён, т.к.
> dashboard сам по себе показывает состояние через `--` метрик и пустой список
> контейнеров. Раздел оставлен для исторического референса по визуальному языку.

**Два состояния:**
- `NO WIFI` — нет подключения к сети вообще
- `NO SERVER` — WiFi есть, сервер не отвечает

---

### Screen 5 — ALERT

**Данные:** `level: "critical"|"warning"`, `message: str` (≤ ~50 символов)

**Текущие проблемы:**
- Полностью заливается красным/оранжевым — текст плохо читается на насыщенном фоне
- Один символ "!" без иконки

**Что хочется:**
- Тёмный фон, яркая цветная полоса/рамка сверху или по периметру
- Читаемый крупный символ предупреждения + текст сообщения в 2–3 строки
- Разница critical (красный акцент) vs warning (янтарный) должна быть очевидна

## Цветовая система

| Состояние | Цвет | Hex |
|---|---|---|
| OK / Healthy | Зелёный | `#00E676` или `#39FF14` |
| Warning / Starting | Янтарный | `#FFB300` |
| Critical / Error | Красный | `#FF3D3D` |
| Unknown / Stale | Тёмно-серый | `#4A5568` |
| Фон | Почти чёрный | `#0A0E14` |
| Текст основной | Белый | `#FFFFFF` |
| Текст secondary | Серый | `#8896A8` |

## Технические ограничения для дизайна

| Ограничение | Причина |
|---|---|
| Нет sub-pixel сглаживания шрифтов | Arduino_GFX рендерит bitmap fonts |
| Нет AA на дугах | Arduino_GFX fillArc не антиалиасирует края |
| Нет плавных градиентов | Дорого по пикселям; лучше flat + дуги |
| Нет touch-элементов | Устройство без ввода |
| Скруглённые углы | Учитывать safe area ≈ 20px от угла квадрата |
| 16-bit цвет (RGB565) | Ограниченная палитра, нет RGBA |
| nearest-neighbor scaling (врем. #2) | Небольшая пикселизация при 480→272 |

---

## Формат результата

Для каждого из 5 экранов:

1. **Mockup 480×480 px** (дизайн под это разрешение; физически выводится на 240×240)
2. **Описание layout-решения** и почему оно лучше текущего
3. **Hex-цвета** для каждого значимого элемента

Mockup должен визуально учитывать скруглённые углы (safe area ≈ 30px от угла).
