# План: display_order для операций в паре (work_type, zone)

## Цель

Зафиксировать порядок операций в каждой паре (work_type, zone) независимо. Сейчас сортировка по `Operation.id` — порядок ломается при добавлении новых операций (предложенных работником через `/complete`).

## Решение

`display_order INTEGER` в junction-таблице `operation_work_type_zone`. Одна и та же операция в разных парах может иметь разный порядок.

**Альтернатива (отвергнута):** `display_order` на самой `operations` — даёт один глобальный порядок, не покрывает кейс «одна операция в двух парах в разных местах».

**Связанные файлы:**
- `backend/svc_tasks/app/domain/models.py:155` — модель `OperationWorkTypeZone`
- `backend/svc_tasks/app/api/operations.py` — `GET /operations`, `GET /operations/pending`
- `backend/svc_tasks/app/api/tasks.py:71` — `_enrich_task` (операции в `GET /{task_id}`)
- `backend/svc_tasks/app/api/tasks.py:919` — создание операции через `/complete`

---

## Принятые решения

1. **Где редактируется порядок:** операции живут только в WFM (не приходят из LAMA). Пока правка `display_order` будет точечно через БД (`UPDATE ... WHERE ...`). API пересортировки откладывается до появления регулярной потребности и UI.
2. **Новые PENDING-операции от работника:** вставляются в общий конец пары — `MAX(display_order)+1`.
3. **`GET /operations/pending`:** глобальный список без пары — сортировка по `id` остаётся (порядок предложения).
4. **Поле в API-ответе:** `display_order` возвращается в `OperationResponse` (полезно для будущего UI и для отладки).

## ADR: display_order на junction-таблице

**Дата:** 2026-05-04
**Режим:** FULL
**Owner:** Pavel Karpychev
**Review trigger:** появление UI редактирования операций (в WFM или на стороне LAMA), импортирующего операции с уже заданным порядком

### Контекст
Операции возвращались по `ORDER BY id`. При добавлении новой PENDING-операции через `/complete` она вставала в конец по id, что иногда нарушало бизнес-логичный порядок. Junction-таблица `operation_work_type_zone` уже выражает связь операции с парой (work_type, zone), и порядок может различаться между парами.

### Решение
`display_order INTEGER` хранится в `operation_work_type_zone` (per пара), не в `operations`. Сортировка по `(display_order, id)`. Новые PENDING получают `MAX(display_order)+1` для своей пары.

### Альтернативы
- **A: `display_order` в `operations`** — отвергнуто: одна операция в двух разных парах не может иметь одного «правильного» порядка одновременно.
- **B: гибрид (default на `operations`, override в junction)** — отвергнуто: две правды о порядке → путаница и расхождения при редактировании; ради экономии одного поля при ручном вводе не оправданно.

### Экономическое/архитектурное обоснование
Junction-таблица уже существует — добавление колонки не меняет архитектуру, только обогащает существующее отношение. API пересортировки не реализуем сейчас: операции живут только в WFM, корректировка `display_order` через `UPDATE` в БД достаточна, пока нет регулярного процесса. Когда появится UI — добавим batch-эндпоинт `PATCH /operations/order`.

### Принятые риски
- Ручная правка `display_order` через БД легко создаёт «дыры» (1, 2, 5, 7) — это допустимо: сортировка работает, дыры не мешают. Заполнение `MAX+1` это сохраняет.
- При импорте операций из внешнего источника (если такая интеграция появится) забудут проставить `display_order` → все встанут в `0`. Mitigation: `MAX+1` в коде импорта, но кода импорта пока нет.

### Future hook
- Поле `display_order: Optional[int]` в `OperationResponse` уже есть — клиент может отображать или скрывать; когда появится UI пересортировки, поле станет обязательным в контексте пары.
- Если потребуется пересортировка через API — batch-эндпоинт `PATCH /operations/order` с body `{work_type_id, zone_id, operation_ids: [...]}` (одна транзакция, атомарность).
