# Δt — длительность управленческого цикла

**Концепт-документ.** Метрика **не реализована** ни в API, ни в дашбордах. Здесь зафиксировано определение, источники данных и место в продукте, чтобы при будущей реализации не пересчитывать дизайн с нуля.

**Связанные документы:**
- `.memory_bank/strategy/theoretical_foundation.md` — теоретический контекст (EIM, Vial DT)
- `.memory_bank/analytics/task_events_backend.md` — аудит-лог `task_events`
- `.memory_bank/domain/task_model.md` — модель задачи

---

## Определение

**Δt** = `task.completed_at` − `task.created_at` для задач с `review_state = ACCEPTED`.

- Берутся **только задачи, прошедшие приёмку**. Незакрытые и отклонённые не учитываются — иначе метрика смешает «скорость» и «качество».
- Для задач с `acceptance_policy = AUTO` `completed_at` ≈ момент `AUTO_ACCEPT`. Для MANUAL — момент `ACCEPT` от управляющего.
- Для LAMA-задач началом цикла считается `created_at` в WFM, не время появления в LAMA. Это разумно для MVP; при необходимости — отдельная метрика «LAMA latency».

---

## Зачем нужна

Vial (2019) определяет ценность цифровой трансформации через **скорость организационного ответа на сигналы среды**. Δt — это операционализация: насколько быстро магазин закрывает обнаруженную ситуацию (выкладка, переоценка, инцидент).

**Дополняет, а не заменяет**, существующий бизнес-кейс «снижение ФОТ»:
- **ФОТ** — финансовый показатель, легко считается, понятен SMB-покупателю
- **Δt** — стратегический показатель, устойчив к ценовому давлению, убедителен для зрелых клиентов

В рамках EIM Δt — прямая измеримая ценность шагов S→T→E→F цикла.

## Источники данных

Все нужные поля **уже есть в БД**, новых таблиц не требуется:

| Поле | Источник |
|---|---|
| `task.created_at` | `tasks` |
| Время `ACCEPT` или `AUTO_ACCEPT` | `task_events` (`event_type IN ('ACCEPT', 'AUTO_ACCEPT')`) |
| `review_state` | `tasks` |
| `work_type_id`, `zone_id`, `assignee_id`, `shift_id` (для группировок) | `tasks` |
| `store_id` (через `assignment`) | JOIN `tasks → assignment → store` |

Базовая выборка: `SELECT task_id, created_at, (event_completed_at - created_at) AS dt FROM tasks JOIN task_events ON ...`. Без обогащения схемы.

## Базовые группировки

Метрика интересна **в разрезе**, не как одно число:

- **По типу работ** (`work_type_id`) — выявляет «медленные» категории
- **По зоне** (`zone_id`) — обнаруживает зональные узкие места
- **По сотруднику** (`assignee_id`) — индивидуальная производительность (с осторожностью, см. ограничения)
- **По магазину** (`store_id`) — сравнение точек
- **По смене** (`shift_id`) — нагрузочные дни vs обычные

**Перцентили важнее среднего:** среднее искажается выбросами (задача висела сутки забытая). Минимум — медиана и P90.

## Ограничения и риски

- **Закон Гудхарта.** Если Δt станет KPI работника, появится gaming: задачи будут закрываться раньше реального завершения. Защита — сравнение с baseline + анализ через `task_events.history_brief.duration` (фактическое IN_PROGRESS время).
- **Отклонённые задачи.** Если задача отклонена и переоткрыта, какой брать `created_at`? **Решение:** оригинальный `created_at` (это и есть Δt управленческого цикла, включая исправление). Альтернатива — отдельная метрика «время до первой попытки приёмки».
- **AUTO-приёмка vs MANUAL.** Δt у AUTO почти всегда меньше — это не повод сравнивать политики напрямую. Группировки делать **внутри политики**.

## Когда реализовывать

**Не сейчас.** Реализация = endpoint `/tasks/analytics/dt` (или встраивание в существующий dashboard) + UI на стороне менеджера. Запускается при появлении конкретной потребности:

- Внутренний запрос: «нужна метрика прогресса проекта, которая не привязана к ФОТ»
- Внешний запрос: первый клиент, спрашивающий «как измерить эффект внедрения, кроме ФОТ»

До этого момента — концепт зафиксирован, данные собираются, реализация — отдельный спринт.
