# Параллельные subagents — overlap check ОБЯЗАТЕЛЕН

Перед запуском любой параллельной волны агентов (`Agent` с `isolation: "worktree"`) — **всегда проверять file overlap** между новым агентом и всеми уже запущенными.

## Чек-лист перед запуском нового агента

1. **Карта активных агентов:** какие файлы/feature-папки сейчас затрагиваются другими агентами. Для каждого: входной файл + cross-references.

2. **Карта нового агента:** какие файлы он тронет.
   - **Split-агент** — трогает один файл + создаёт sub-folder. Изолированно, low risk.
   - **Dedup-агент** — трогает ВСЕ feature-файлы которые используют дублируемый компонент. High risk пересечений.

3. **Сравнить.** Есть пересечения → переключить на другой безопасный файл или ждать.

## Безопасные паттерны параллелизма

- **Splits в разных feature-папках** — почти всегда безопасны
- **Один dedup + splits в нетронутых им features** — безопасно
- **Splits внутри одной папки на разных файлах** — безопасно (если файлы не импортируют друг друга)

## Опасные паттерны (НЕ запускать параллельно)

- **2+ dedup-агента одновременно** — почти всегда есть overlap по features
- **Split + dedup в той же feature-папке** — split трогает файл, dedup тоже его обновит
- **Любой агент работает в `components/shared/` пока другой добавляет туда новые компоненты** — конфликт по `index.ts` re-exports

## Если overlap неизбежен

Запускать **sequentially**: один → merge в main → второй. Не пытаться "наплевать, разрулим конфликты" — это съест больше времени чем серийный запуск.

## После merge каждого агента

`git pull` в worktree'ы остальных активных агентов чтобы они работали поверх свежего main и не накапливали drift.

## Лимит

До 6 параллельных агентов одновременно (см. `feedback_split_pattern` в auto-memory). Чем ближе к лимиту, тем выше риск что хотя бы у двух будет overlap — снижай количество.
